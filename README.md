# quay-aggregator-integrations

Aggregator adapters for the Quay program, packaged as a standalone repo
for integrator teams. Each adapter is its own crate; two vendored Quay
crates make the repo build with no external Quay sources.

| Crate | Path | Purpose |
| --- | --- | --- |
| `quay-aggregator-titan` | `titan/` | `TradingVenue` impl for routers using `titan-integration-template` |
| `quay-aggregator-jupiter` | `jupiter/` | `Amm` impl for routers using `jupiter-amm-interface 0.6` |
| `quay-sdk` | `vendor/quay-sdk` | Off-chain client: instruction builders, PDA derivation, zero-copy account decoders, swap simulator |
| `quay-vm` | `vendor/quay-vm` | Pricing VM the simulator runs — the same interpreter executes inside the on-chain `swap` ix, so off-chain quotes are bit-identical by construction |

Consume an adapter as a git dependency; cargo locates the package inside
the workspace:

```toml
quay-aggregator-titan = { git = "<this repo>" }
# or
quay-aggregator-jupiter = { git = "<this repo>" }
```

Each adapter instance represents **one Strategy** — a pricing curve bound
to one `(base_mint, quote_mint)` pair on a MarketMaker. Routers typically
hold many (one per active strategy). See each adapter's own README for
trait-level details.

## Build

The jupiter adapter is its own workspace with its own `Cargo.lock`:
`jupiter-amm-interface 0.6` pins `solana-instruction = 2.2.1` exactly,
which conflicts with the 2.3.x line the titan template graph resolves.
Everything else (titan + vendored crates) shares the root workspace.

```bash
cargo build && cargo test --workspace   # titan + vendored crates
cd jupiter && cargo build && cargo test # jupiter
```

The titan adapter's `titan-integration-template` dependency is
`edition = "2024"`, so rustc >= 1.85 is required.

## On-chain parity tests

Each adapter ships `tests/parity.rs`, driven by the fixtures in
`fixtures/*.json`. The fixtures were generated by executing real swaps
against the compiled Quay program (in a LiteSVM): each one carries the
full account set an integrator fetches over RPC, plus
`(side, amount_in, actual amount_out)` for an amount sweep on both sides
— `out: null` rows are swaps the program itself refused.

The tests rebuild the venue from nothing but those raw account blobs —
the exact `from_account` / `from_keyed_account` → `update` → `quote`
flow a router runs — and assert `quote()` reproduces the on-chain
outcome for every row: successful swaps must match the paid amount
exactly, and failed swaps must be refused. The sweep covers flat,
spread-with-protocol-fee, exact-fee-calibrated, and one-sided-reject
curves, plus amounts that overdraw the maker's inventory.

Fixtures are regenerated by the Quay team from program sources; treat
them as ground truth for the deployed program version.

## Vendored crates

`vendor/quay-sdk` and `vendor/quay-vm` are snapshots of the corresponding
Quay crates, trimmed to what the adapters need (DSL authoring/template
modules and internal test suites are not included). They are updated by
re-vendoring from the Quay source tree — don't patch them here; report
issues to the Quay team instead.
